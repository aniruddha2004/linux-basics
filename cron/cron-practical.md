# Cron Practical Walkthrough

> Step-by-step guide to creating scripts, testing them with `run-parts`, and scheduling them with cron.

---

## Overview

This walkthrough covers the full lifecycle of a cron job:
1. Create scripts with proper naming
2. Add shebang and make executable
3. Test them with `run-parts`
4. Schedule via `crontab -e`
5. Verify output

---

## Step 1: Create Scripts

```bash
mkdir ~/test-scripts
cd ~/test-scripts
```

Create two scripts:

```bash
vi script1
```

```bash
#!/bin/bash
echo -n "Hi there!"
```

```bash
vi script2
```

```bash
#!/bin/bash
echo -n "Welcome to the world of crons!!!"
```

> **Note**: We name them `script1` and `script2` (no `.sh` extension) because `run-parts` ignores any file containing a dot in its name. Files like `script1.sh` or `backup.old` are silently skipped.

> **Note**: The `#!/bin/bash` line (shebang) at the top is mandatory. It tells the kernel which interpreter to use when executing the file.

---

## Step 2: Make Them Executable

```bash
chmod +x script1
chmod +x script2
```

> `run-parts` only runs files that have execute permission. Without `chmod +x`, the file will be silently ignored.

---

## Step 3: Test with `run-parts`

### Check Which Scripts Are Eligible

```bash
run-parts --list ~/test-scripts
```

```text
test-scripts/script1
test-scripts/script2
```

> `--list` shows which files `run-parts` considers eligible. A file must: (1) have no dots in its name, (2) have execute permission, (3) not be a backup file.

### Dry Run (Preview Without Executing)

```bash
run-parts --test ~/test-scripts/
```

```text
test-scripts/script1
test-scripts/script2
```

> `--test` shows what would execute without actually running anything.

### Execute and See Output

```bash
run-parts --report ~/test-scripts/
```

```text
test-scripts//script1:
Hi there!test-scripts//script2:
Welcome to the world of crons!!!
```

> `--report` prefixes each script's output with its path so you know which script produced what.

---

## Step 4: Schedule with Cron

### Create the Crontab Entry

```bash
crontab -e
```

Add this line at the bottom:

```cron
*/2 * * * * run-parts --report /home/aniruddha_ghosh/test-scripts >> /tmp/testing-cron.txt
```

| Part | Meaning |
|------|---------|
| `*/2 * * * *` | Every 2 minutes |
| `run-parts --report ...` | Execute all scripts in the directory |
| `>> /tmp/testing-cron.txt` | Append output to a log file |

Save and exit. You will see:

```text
crontab: installing new crontab
```

### Verify the Crontab

```bash
crontab -l
```

Shows your installed cron jobs.

---

## Step 5: Check Output

Wait a few minutes, then check the log:

```bash
cat /tmp/testing-cron.txt
```

```text
/home/aniruddha_ghosh/test-scripts/script1:
Hi there!/home/aniruddha_ghosh/test-scripts/script2:
Welcome to the world of crons!!!/home/aniruddha_ghosh/test-scripts/script1:
Hi there!/home/aniruddha_ghosh/test-scripts/script2:
Welcome to the world of crons!!!
```

Each execution appends to the file. You can see the scripts ran twice (after 2 minutes and 4 minutes).

---

## Step 6: Inspect the Stored Crontab

Cron saves user crontabs here:

```bash
cat /var/spool/cron/crontabs/$(whoami)
```

Or as root:

```bash
sudo cat /var/spool/cron/crontabs/aniruddha_ghosh
```

```text
# DO NOT EDIT THIS FILE - edit the master and reinstall.
# (/tmp/crontab.0gIjKp/crontab installed on Mon Jun 22 05:39:42 2026)
# ... (header comments) ...
*/2 * * * * run-parts --report /home/aniruddha_ghosh/test-scripts >> /tmp/testing-cron.txt
```

> **Warning**: Never edit this file directly. Always use `crontab -e`. Cron manages this file and your direct edits may be overwritten.

## Requirements for `run-parts`

For a script to be executed by `run-parts`, all of these must be true:

| Requirement | Why | How to Check |
|-------------|-----|--------------|
| **No dots in filename** | `run-parts` ignores files with dots | `script1` works, `script1.sh` is ignored |
| **Execute permission** | The file must be runnable | `chmod +x script1` |
| **Valid shebang** | The kernel needs to know the interpreter | `#!/bin/bash` as first line |
| **Not a backup file** | Files ending in `~` or `.bak` are ignored | Avoid `script1~` or `script1.bak` |

---

## Best Practices

1. **Always use full paths** in cron commands — cron has a minimal PATH environment
2. **Redirect output** (`>> /var/log/myjob.log 2>&1`) to capture errors
3. **Test scripts manually first** before putting them in cron
4. **Use `run-parts --test`** to preview what would run
5. **Use `run-parts --report`** to see which script produced which output
6. **Check `/var/log/syslog`** or `/var/log/cron` for cron execution logs

---

## Quick Commands

| Command | Purpose |
|---------|---------|
| `crontab -e` | Edit your crontab |
| `crontab -l` | List your crontab |
| `crontab -r` | Remove your crontab |
| `run-parts --list dir` | Show which scripts would run |
| `run-parts --test dir` | Dry-run (show but don't execute) |
| `run-parts --report dir` | Run and prefix output with script name |
| `sudo systemctl status cron` | Check if cron daemon is running |
